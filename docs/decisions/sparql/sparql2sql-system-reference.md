# SPARQL→SQL System Reference (Current Implementation)

Date: 2026-04-10  
Scope: `insurance` dataset, `/query/vkg/translate`, Ontop-backed translation flow

## 1. What This System Does Today

The system translates SPARQL into SQL using Ontop, then returns SQL text through:

- `POST /query/vkg/translate?dataset=insurance&projected={true|false}`

Primary implementation:

- Controller: `services/backend/src/main/java/com/unconcealment/backend/controller/QueryController.java`
- Translator: `services/backend/src/main/java/com/unconcealment/backend/service/OntopVkgService.java`

Response shape:

```json
{
  "sql": "SELECT ...",
  "projectedRequested": true,
  "projectedApplied": true
}
```

Meaning:

- `projected=false`: raw Ontop SQL (may include helper columns)
- `projected=true`: SQL wrapped to project output columns to SPARQL answer variables

## 2. Artifact Roles (How Files Work Together)

### 2.1 Postgres tables (source of truth data)

- Data lives in Postgres schema `acme_insurance`.
- SQL generated by Ontop executes against these tables.

### 2.2 `core.ttl` (ontology vocabulary)

Path:

- `ontology/insurance/core.ttl`

Purpose:

- Defines classes and properties in the insurance vocabulary (`in:` namespace).
- SPARQL queries use this vocabulary.
- Does not define SQL joins by itself.

### 2.3 `ontop.obda` (critical mapping layer)

Path:

- `ontology/insurance/ontop.obda`

Purpose:

- Maps ontology terms to relational tables/columns/joins.
- This is the file Ontop uses at runtime to rewrite SPARQL into SQL.
- If table/column mappings are missing here, SPARQL cannot translate correctly.

How it works conceptually:

1. Target: RDF triple pattern template (`in:Policy`, `in:policyNumber`, etc.)
2. Source: SQL query/table expression exposing required columns
3. Ontop rewrites SPARQL algebra into SQL by composing these mappings

### 2.4 `bindings.ttl` (R2RML-style mapping documentation/graph)

Path:

- `ontology/insurance/bindings.ttl`

Purpose:

- Semantic binding representation used in the RDF side (`urn:insurance:bindings`).
- Useful for reasoning/introspection/documentation workflows.
- Current translation endpoint is powered by Ontop + `ontop.obda`; `bindings.ttl` is not the direct runtime translator.

### 2.5 `acme-benchmark.ttl` (benchmark workload definition)

Path:

- `data/cwd-benchmark-data/ACME_Insurance/investigation/acme-benchmark.ttl`

Contains:

- NL prompt
- SPARQL query
- expected SQL query

Used by:

- `services/backend/src/test/java/com/unconcealment/backend/OntopBenchmarkTest.java`

## 3. End-to-End Request Flow (`/query/vkg/translate`)

1. Receive SPARQL + `dataset` + `projected`.
2. Normalize SPARQL in `OntopVkgService`:
   - strip benchmark `SERVICE ... mapped` wrapper
   - ensure needed prefixes (`rdf`, `xsd`)
   - normalize bare date literal formats used in benchmark SPARQL
3. Build executable Ontop query.
4. Extract native SQL from Ontop `NativeNode`.
5. If `projected=true`, wrap SQL so output columns match SPARQL variable projection.
6. Return SQL JSON response.

## 4. How to Onboard New SQL Tables

When adding a new table to the domain, do all layers below.

1. Database layer
- Create table and constraints (PK/FK) in Postgres.
- Load data.

2. Ontology layer (`core.ttl`)
- Add class/property terms only if new domain concepts are introduced.
- Reuse existing terms if concept already exists.

3. Mapping layer (`ontop.obda`) — required
- Add class mapping (`rdf:type` equivalent) for entity rows.
- Add data property mappings for new columns.
- Add object property mappings for FK relationships/joins.

4. Optional RDF binding layer (`bindings.ttl`)
- Keep synchronized if your team uses it for graph introspection.

5. Validation
- Restart backend (Ontop mappings load at startup).
- Call endpoint for representative SPARQL.
- Run benchmark/integration tests.

## 5. How Changes Propagate (SQL → Ontology Annotations)

### 5.1 New column on existing table

1. Add column in SQL schema.
2. Decide semantic meaning:
   - existing property or new property.
3. If new property, update `core.ttl`.
4. Add/adjust mapping in `ontop.obda` to expose that column to SPARQL.
5. Update benchmark or test queries if this column is now required output/filter logic.

### 5.2 Column rename/type change

1. Update SQL schema.
2. Update all affected `ontop.obda` source SQL fragments.
3. Verify casts/aggregates still valid in generated SQL.
4. Re-run endpoint comparison and benchmark.

### 5.3 New relationship (FK)

1. Add FK in SQL schema.
2. Add object property mapping in `ontop.obda` for join traversal.
3. Add property declaration in `core.ttl` if new predicate.
4. Validate with SPARQL that traverses the new relation.

## 6. What to Do for Large Schema Expansions

Treat as a mapping migration project.

1. Schema diff
- Produce old/new table+column+FK diff.

2. Batch mapping update plan
- Batch A: entity class mappings
- Batch B: data property mappings
- Batch C: object property/FK mappings
- Batch D: aggregate/date-heavy query validation

3. Automated verification
- Benchmark test:
  - `cd services/backend && mvn test -Dtest=OntopBenchmarkTest -Dgroups=integration`
- Endpoint vs benchmark SQL check:
  - `scripts/compare_vkg_translate_vs_csv.sh`
- Endpoint output export:
  - `target/ontop-endpoint-output.csv`

4. Regression triage order
- Translation failures (HTTP 500)
- `projectedApplied=false` where it was expected true
- SQL mismatch
- result mismatch

## 7. Benchmark and Viewer: How to Read Them

### 7.1 Benchmark output CSV

Path:

- `target/ontop-benchmark-output.csv`

Key columns:

- `generated raw sql`
- `generated projected sql`
- `projected applied`
- `generated raw result`
- `generated projected result`
- `expected result`

Interpretation:

- raw result mismatch can be acceptable if helper columns exist.
- projected result should match expected result for pass cases.

### 7.2 Endpoint output CSV

Path:

- `target/ontop-endpoint-output.csv`

Purpose:

- Captures live endpoint output per benchmark SPARQL.
- Includes endpoint status/errors to detect runtime divergence.

### 7.3 Astro viewer

Page:

- `services/web/src/pages/benchmark/sparql2sql.astro`

Use:

- View long SQL/SPARQL/results side-by-side in browser.
- Filter by query text.

## 8. Current Known Limitations

1. Ontop unsupported functions can fail translation.
- Example: `date_diff` expression in one benchmark case.

2. Local DATE_DIFF support mismatch.
- Benchmark test currently skips one DATE_DIFF-related case on local Postgres.

3. Mappings are startup-loaded.
- Changes in `ontop.obda` require backend restart.

## 9. Minimal Operator Checklist

When changing schema/mappings:

1. Update SQL schema/data.
2. Update `ontop.obda` (mandatory).
3. Update `core.ttl`/`bindings.ttl` as needed.
4. Restart backend.
5. Run benchmark + endpoint comparison.
6. Inspect CSV/viewer for regressions before merge.
